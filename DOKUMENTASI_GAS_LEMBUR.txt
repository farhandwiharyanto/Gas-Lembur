# Strict Role Provisioning & Login Restriction

Implementasi manajemen akses ketat di mana Admin hanya memasukkan Inisial karyawan untuk memberikan *Role*, sementara Karyawan yang belum diberi *Role* sama sekali tidak diizinkan masuk ke aplikasi.

## Proposed Changes

### Admin User Management UI
#### [MODIFY] resources/views/admin/users/index.blade.php
- Menghapus form input manual untuk *Nama Lengkap*, *Email*, dan *Password*.
- Menyisakan hanya input **Username (Inisial)** dan **Dropdown Role**.
- Tampilan tabel tetap menampilkan *User* yang sudah terdaftar di database lokal.

### Admin Provisioning Logic
#### [MODIFY] app/Http/Controllers/UserController.php
- Membatasi validasi hanya untuk `username` dan `role`.
- Controller akan mengambil data karyawan aslinya **langsung dari database SNI** menggunakan `LdapService`.
- Jika Inisial karyawan tidak ada di SNI, akan muncul *error*: "Karyawan dengan inisial tersebut tidak ditemukan di SNI."
- Jika berhasil, data *Nama* dan *Email* akan diisi otomatis dari hasil _query_ SNI tersebut, kemudian disimpan ke tabel lokal `users` beserta *Role*-nya.

### Login Restriction
#### [MODIFY] app/Http/Controllers/AuthController.php
- Menambahkan pengecekan *database lokal* persis **sebelum** proses LDAP (baris ke-38).
- Jika `User` dengan inisial tersebut **belum didaftarkan oleh Admin**, sistem akan langsung menolak dan mencetak pesan: "Maaf, akun Anda belum memiliki hak akses / Role. Silakan hubungi Admin."

### LDAP Fallback Removal
#### [MODIFY] app/Services/LdapService.php
- Mengubah fungsi `private data_sni()` menjadi `public` agar bisa dipanggil oleh `UserController` di atas.
- Menghapus _logic_ `User::create(...)` di baris ke-103. Setelah perubahan ini, proses koneksi sukses LDAP hanya akan meng-_update_ data Sinkronisasi SNI tanpa bisa membuat user baru. Karyawan harus sudah berstatus "terdaftar".

## Verification Plan

### Manual Verification
Pekerjaan ini akan Anda validasi secara manual dengan cara:
1. Login sebagai `adm`.
2. Masuk ke halaman **Management Pengguna**. Form baru hanya akan menampilkan Inisial dan Role.
3. Masukkan Inisial sembarang yang salah (misal "XXX"), sistem harus menolak karena tidak ada di SNI.
4. Masukkan Inisial Anda sendiri beserta Role tertentu. Sistem akan sukses mendaftarkan dan menarik otomatis nama panjang Anda dari SNI.
5. Lakukan Logout.
6. Coba login menggunakan Inisial teman Anda yang *belum didaftarkan* Admin (tapi password LDAP-nya benar). Sistem harus menolak loginnya.
7. Login ulang menggunakan Inisial Anda sendiri yang *tadi telah didaftarkan*. Anda akan masuk dengan mulus.

# Phase 7: Strict Role Provisioning Walkthrough

### Changes Implemented
Sistem manajemen hak akses telah diperketat sepenuhnya sesuai instruksi Anda. Proses manual ketik-mengetik nama dan email telah dihapus untuk meminimalkan *human error* dan memaksimalkan Source of Truth dari SNI.

#### 1. Admin Provisioning Otomatis
Pada menu **Management Pengguna**, form penambahan _user_ baru sudah disederhanakan.
- **Kondisi Sebelum**: Admin harus mengetik Nama, Email, dan Password.
- **Kondisi Sekarang**: Admin hanya mengisi 2 field:
  - **Username (Inisial)**
  - **Role/Peran** (User / Pimpinan / Admin)

**Alur Kerja Form Baru:**
Saat Admin memencet tombol "Simpan Pengguna", sistem akan pergi ke _database_ SNI (melalui `LdapService::data_sni()`) untuk memvalidasi apakah Inisial tersebut benar-benar ada. 
- Jika **Tidak Ada** di SNI: Form akan ditolak dan memunculkan *error* "Inisial / Karyawan tidak ditemukan di sumber data SNI".
- Jika **Ada** di SNI: Sistem secara gaib (otomatis) menarik Nama Lengkap, Email Lengkap, dll dan menyelipkannya ke tabel `users` lokal berdampingan dengan Role yang Anda pilih tadi.

#### 2. Kunci Pintu Login (Login Restriction)
- File `AuthController.php` telah dimodifikasi pada baris awal logika LDAP.
- Sekarang, bahkan sebelum sistem mencoba menghubungi (*bind*) Active Directory Lintasarta, sistem memeriksa _database_ lokal Anda: *"Apakah Inisial ini ada di tabel `users`?"*
- Jika **Belum Ada** (artinya Admin belum mendaftarkan dan memberinya Role), sistem langsung memblokir login dalam sepersekian detik dengan peringatan: *"Maaf, Inisial akun Anda belum didaftarkan untuk mendapatkan Role akses. Silakan hubungi Administrator."*

#### 3. Pembersihan Auto-Provisioning
- Di versi sebelumnya, setiap orang yang sukses masuk LDAP akan dibuatkan akun otomatis dengan Role 'user'. Fungsi tersebut sekarang sudah dicabut 100%. Hanya Admin yang punya kuasa mendaftarkan (*provisioning*).

### Validation Results
Silakan pastikan perubahan ini berjalan sebagaimana mestinya di laptop Anda:
1. Pastikan Anda **Login sebagai `adm`**.
2. Masuk ke **Management Pengguna**, Anda hanya akan melihat form input Inisial. Cobalah masukkan inisial dan Role kolega Anda.
3. Logout as `adm`.
4. Minta kolega tersebut untuk mencoba Login di layar awal menggunakan Inisial mereka masing-masing. Mereka baru bisa masuk jika Anda sudah me-registrasi mereka di langkah 2!

BAGIAN 4: RIWAYAT KENDALA (GAGAL/ROLLBACK)
================================================================================

Selama siklus pengembangan aplikasi Gas-Lembur, terdapat beberapa pendekatan instruksi yang sempat gagal atau sengaja di-rollback karena tidak sesuai kondisi lapangan:

1. [GAGAL/ROLLBACK] Autentikasi LDAP (Active Directory)
   - Kendala: Proses bind LDAP memakan waktu lebih dari 30 detik sehingga menyebabkan error "Fatal error: Maximum execution time of 30+2 seconds exceeded in LdapService.php". Sinkronisasi ke domain `ladomain.lintasarta.co.id` kemungkinan besar terkendala network/VPN/konfigurasi port.
   - Tindakan: Fitur LDAP di-rollback untuk sementara waktu. Login dikembalikan menggunakan autentikasi *lokal* (menggunakan tabel `users` SQLite) tetapi logika verifikasi tetap dibuat berbasis SNI dan *Username* diganti menjadi format "Inisial 3 Huruf".

2. [GAGAL/ROLLBACK] Pembuatan Akun Otomatis (Auto-Provisioning) & Bentrok Email
   - Kendala: Muncul error "SQLSTATE[23000]: Integrity constraint violation: 19 UNIQUE constraint failed: users.email" saat menguji skenario login akun yang belum terdaftar. Fungsi auto-provisioning sebelumnya menabrak *rule* database SQLite karena data asalnya bentrok.
   - Tindakan: Sistem *auto-mendaftar* secara otomatis dicabut (*Strict Provisioning*). Saat ini **semua** otorisasi dikelola ketat oleh Admin. Admin cukup menginput Inisial, lalu sistem menarik detail SNI perlahan di awal untuk menghindari bentrok data.

3. [GAGAL/ROLLBACK] Input Tanda Tangan Menggunakan "Draw Pad / Canvas"
   - Kendala: Menggunakan canvas digital dirasa kurang praktis untuk Karyawan dan Pimpinan karena menghambat *workflow* (susah menggambar di layar bagi pengguna PC tanpa mouse external/trakpad).
   - Tindakan: Fitur di-rollback secara utuh ke formulir **Upload File Gambar (.jpeg / .png)** klasik yang jauh lebih *user-friendly*!

4. [GAGAL/ROLLBACK] Otoritas Ganda pada Menu Admin
   - Kendala: Di pertengahan jalan, halaman *Data Lembur* bagi Admin direncana bisa memiliki fitur Setujui/Tolak. Namun ini diprotes karena mengakibatkan tumpang tindih otorisasi; validasi aktual wajib ditutup murni via Pimpinan departemen saja.
   - Tindakan: Kolom Aksi dilepas dan dicabut dari *Admin Table*, menggabungkan data menjadi arsip *read-only* bagi Admin.

5. [FIX ISSUES] SQLite Compatibility Error (Fungsi FIELD)
   - Kendala: Sempat terdapat error "General error: 1 no such function: FIELD" pada *view* riwayat Pimpinan karena `ORDER BY FIELD()` adalah fungsi bawaan *MySQL*, sedangkan environment Laravel Anda menggunakan database *SQLite*.
   - Tindakan: Diperbaiki otomatis menggunakan metode query konvensional dengan blok logika SQL asli Laravel (`CASE WHEN THEN`).

BAGIAN 6: PANDUAN PENGELOLAAN GITHUB (REMOTE, PUSH, PULL)
================================================================================

Berikut tata cara resmi untuk menyambungkan, melakukan Push, maupun Pull project Gas-Lembur ke repository GitHub Anda (Akun: farhandwiharyanto):

### 1. Inisialisasi Lokal (Cara Menyambungkan ke GitHub)
Apabila folder ini belum terinisialisasi secara remote Git, jalankan command ini sekali saja:
```bash
git init
git add .
git commit -m "First commit: Gas-Lembur Initial Release"
git branch -M main
git remote add origin https://github.com/farhandwiharyanto/Gas-Lembur.git
git push -u origin main
```
*Catatan Penting: Pastikan Anda terlebih dahulu membuat repository/wadah kosong bernama "Gas-Lembur" di halaman web GitHub Anda.*

### 2. Tata Cara Melakukan Push (Menyimpan Perubahan Harian)
Setiap kali Anda selesai memodifikasi kode, memperbaiki bug, atau memperbarui dokumentasi (seperti file .txt ini), Anda wajib melempar kode tersebut ke awan (GitHub):
```bash
git add .
git commit -m "Pesan update kerjaan hari ini, contoh: Fix fitur pimpinan"
git push origin main
```

### 3. Tata Cara Melakukan Pull (Menarik Kode Terkini)
Saat Anda berganti laptop kerja atau rekan kerja melakukan Push ke repository yang sama, Anda perlu menarik pembaruan terbaru agas selaras:
```bash
git pull origin main
```

*Tips:*
Jika ditanya _username_ dan _password_ di Terminal saat melakukan proses push/pull, masukkan Username GitHub Anda dan masukkan **Personal Access Token (PAT)** sebagai password.*
